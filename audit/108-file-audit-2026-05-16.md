# Omerion Agentic Workflow Factory — Zero-Mercy Audit

**Date:** 2026-05-16
**Scope:** 108-file build across `core/`, `api/`, `departments/`, `n8n/`, `industry_packs/`, `clients/`, `docker/`, `railway.toml`, `main.py`
**Method:** Four parallel read-only audits over the entire scope, then synthesis. Findings cite `file:line` where available.
**Bottom line:** Not deploy-ready. The headline subsystem (RSI self-improvement) is theater. HITL approval does not actually resume workflows. Worker scheduling will not survive a Railway restart. ~6 of 26 agents are deterministic logic in agent costume.

---

## PART 1 — Architectural Alignment

### Goal 1 — Universal Agentic Workflow Factory: **PARTIAL FAIL**

A new client *can* be onboarded by editing YAML — `config_loader.py` cleanly merges `industry_pack.yaml` + `client_config.yaml`. But the universality is shallow.

Evidence:
- `core/runtime/config_loader.py:81-90` — loader hardcodes the names of real-estate sub-files (`personas.yaml`, `qualification.yaml`, `scoring.yaml`, `sources.yaml`, `rag_queries.yaml`). Missing files in a non-RE pack silently degrade to empty dicts; no error.
- `core/runtime/config_loader.py:137` — `client.industry_pack` value is loaded but **never validated** against the set of available packs at startup. A typo fails at first agent run, not at deploy time.
- `core/runtime/config_loader.py:44-52` — `_deep_merge` only shallow-copies the top dict. Nested overlays will be mutated by callers, leaking cross-client state.
- `clients/_template/client_config.yaml` is **out of sync** with `clients/omerion-internal/client_config.yaml`: the template lacks `discord_channels` under `integrations`. New clients copied from template will silently be missing this.
- `core/schemas/base.py` — `Department` enum is closed-set with 5 values (AGENTIC_FACTORY, LEAD_GEN, etc.). Adding a SaaS-only `BILLING_AUTOMATION` department requires a code change.
- `industry_packs/real_estate/personas.yaml` — 9-tier persona pyramid baked in with role names like "Brokerage Owner", KPIs like `gci_per_agent`. A SaaS pack has to redefine the entire structure with no schema versioning.

**Where it breaks for a non-RE client:** any pack omitting `personas.yaml` results in empty persona context being silently fed to ICP scoring and outreach prompts, which still run and still emit confidence ≥0.6 — the system will look like it works while producing garbage.

**Fix:** (1) Validate `client.industry_pack ∈ available_packs` at app boot, fail-fast. (2) Add `industry_pack.schema_version` and reject mismatches. (3) Make the personas/scoring sub-files optional with explicit `present?` checks in the agents that consume them. (4) Add a script that diffs `_template/client_config.yaml` against the schema in `ClientConfigSpec` and CI-fails on drift.

### Goal 2 — Webhook-First Runtime: **PARTIAL**

The API surface exists and is broad, but several agent actions can ONLY be reached as a side-effect of another endpoint — there is no flat `/agents/{name}/run` entry for downstream agents in a chain.

Public endpoints (`api/webhooks/router.py`, `api/approvals/router.py`):
- `GET /api/v1/health`
- `POST /api/v1/webhooks/lead/intake` — triggers `enrich` → `icp_scorer` synchronously in the request
- `POST /api/v1/agents/{name}/run` — direct agent runner
- `POST /api/v1/departments/{name}/dispatch`
- `POST /api/v1/workflows/{thread_id}/resume`
- `POST /api/v1/webhooks/onboarding/tally` — `aria` → `gatekeeper` → `forge` chain
- `POST /api/v1/webhooks/discord/voice`
- `POST /api/v1/webhooks/rsi/trigger`
- `POST /api/v1/n8n/callback/{recipe}`
- `POST /api/v1/approvals/{approval_id}/decide`

Agent actions only reachable from inside Python (no independent webhook retry):
- `icp_scorer` — only fires after `enrich` succeeds in `/webhooks/lead/intake`
- `gatekeeper` — only inside the tally chain
- `forge` — only inside the tally chain after gatekeeper passes

The `/agents/{name}/run` endpoint *technically* covers these (`api/webhooks/router.py:59-71`), but it bypasses the orchestrated chain context (no upstream payload, no event continuity). It's an escape hatch, not a real retry path.

Secondary defect: `api/webhooks/router.py:36-55` runs the enrich→score chain **synchronously in the HTTP request**. With Hunter API timeout 10s + Claude SONNET ~2-3s, lead intake will hit gateway/proxy timeouts under load.

**Fix:** Convert `/webhooks/lead/intake` to return `202 Accepted` + enqueue (n8n or in-process task). Document the canonical re-trigger path for each chain step.

### Goal 3 — n8n as Deterministic Bridge: **PASS (with one duplication bug)**

All 5 recipes are valid n8n workflow JSON (have `nodes`, `connections`, settings). All URLs are parameterized via `{{$env.OMERION_PUBLIC_BASE_URL}}` etc. — no localhost/example.com hardcoding. HMAC verification is real and uses `hmac.compare_digest` (`core/runtime/n8n_bridge.py:78-80`).

Recipes verdict:

| Recipe | Status | Purpose |
|---|---|---|
| `tally_onboarding_intake.json` | OK | Tally → normalize → POST `/webhooks/onboarding/tally` |
| `lead_sheet_sync.json` | OK | 10-min cron, reads Google Sheet, posts new leads |
| `approval_discord_relay.json` | OK | Inbound webhook → Discord HITL card with approve/reject URLs |
| `rsi_weekly_trigger.json` | **DUPLICATE** | Cron Mon 14:00 UTC → POST `/webhooks/rsi/trigger` |
| `crm_write.json` | OK | Inbound webhook → append to Google Sheet |

Critical issue: `rsi_weekly_trigger.json` AND `core/runtime/worker_main.py:51-68` both schedule the weekly RSI run at Monday 14:00 UTC. There is **no flag, lock, or env switch** to designate one as primary. If both services are deployed, RSI runs twice every Monday — and since the loop is non-idempotent (opens GitHub PRs), this means duplicate PRs.

Two more issues:
- `core/runtime/n8n_bridge.py` has **no dry-run / mock mode**. Local dev requires a live n8n instance or every test fails.
- `api/webhooks/router.py:164` skips HMAC verification when `bridge.configured` is False. Empty `N8N_HMAC_SECRET` in production silently opens all `/n8n/callback/*` endpoints. No startup warning.

**Fix:** Add `ENABLE_WORKER_SCHEDULER` env (default off, since n8n is the source of truth). Add `N8N_DRY_RUN`. Fail startup if `RUNTIME_ENV=prod` and `N8N_HMAC_SECRET` is empty.

### Goal 4 — Recursive Self-Improvement Closed Loop: **FAIL**

The RSI subsystem is theater. The agents run, Claude is called, JSON is produced, Supabase rows are inserted, GitHub PRs are sometimes opened — but the loop **does not close**.

Trace of the actual flow (`departments/recursive_self_improvement/rsi_workflow.py`):
1. `_node_observer` (line 155) queries `agent_runs` table for regressions — **works deterministically, no Claude**.
2. Lines 156-164 run `ux_reviewer`, `token_optimizer`, `rag_auditor`, `seeker` in parallel.
3. Line 165 runs `prompt_optimizer` per flagged agent.
4. Line 166 runs `synthesis` — Claude picks top 3 actions.
5. Line 167 runs `patcher` — opens GitHub PRs IF the action carries `file_path` + `new_content`.
6. Line 185-192: emits `CoreEventType.IMPROVEMENT_PROPOSED` and returns.

Three breaks in the chain:

**Break 1 — Schema mismatch silently drops most actions.**
`prompt_optimizer.py:75-94` returns an `ImprovementProposal` dataclass with `rationale`, `expected_delta`, `confidence` — **no `file_path` and no `new_content`**. Synthesis is asked to produce `patch_payload`, but Claude is free-form and frequently returns `null` or partial payloads. `rsi_workflow.py:111` then filters: `if not payload.get("file_path") or "new_content" not in payload: continue`. Net result: most synthesized actions never reach the patcher.

**Break 2 — No subscriber for `IMPROVEMENT_PROPOSED`.**
A grep across the codebase finds zero `bus.subscribe(CoreEventType.IMPROVEMENT_PROPOSED, ...)`. The terminal event is a log line.

**Break 3 — No post-merge hook.**
`patcher.py:28-99` opens a PR via the GitHub Contents API. A human merges it. There is **no webhook from GitHub back to Omerion to reload configs or restart workers**. `config_loader.py` uses `@lru_cache` with mtime invalidation — but the running container's filesystem doesn't change when a PR merges in the repo. Improvements only take effect on the next deploy.

Additionally:
- `rsi_workflow.py:121-149` — LangGraph use is optional with a sequential fallback. **No `PostgresSaver` is configured in either path.** RSI runs are entirely ephemeral; a crash mid-run loses everything.
- 4 of 7 "RSI agents" make no Claude call: `observer`, `token_optimizer`, partially `rag_auditor`, and structurally `gatekeeper` (in the factory) — they're SQL rollups with hardcoded confidence scores (e.g., `observer.py:99` always returns 0.95).

**Fix:** (1) Tighten the `ImprovementProposal` schema to *require* `file_path` and `new_content` when `kind=PROMPT`, validated before synthesis returns. (2) Add a GitHub webhook handler that triggers a config-cache invalidation + worker bounce. (3) Subscribe to `IMPROVEMENT_PROPOSED` with a HITL-aware patch-application worker, OR delete the event emission entirely and stop pretending. (4) Attach a `PostgresSaver` to the LangGraph compile and to the sequential fallback's state dict.

### Goal 5 — 26 Distinct Agents: **FAIL**

The headcount is fluffed. Roughly 6 of 26 are deterministic functions in agent costume; 4-5 pairs are structurally identical templates differing only by a system prompt string.

Full table appears in [Part 3](#part-3--agent-headcount-justification). Headline overlaps:
- `mapper` + `enrich` — mapper is a one-shot config step misclassified as a per-lead agent.
- `nurture` + `outreach` — same prompt family, same shape.
- `success_ops` + `proposer` — sequential outputs of the same human-facing workflow.
- `strategist` + `librarian` — both summarize tagged signals.
- `sentinel` + `icp_scorer` + `attribution` + `token_optimizer` + `gatekeeper` + `observer` — pure deterministic logic, no Claude call, hardcoded confidence.

Roughly two-thirds of the `_execute()` methods across `departments/lead_gen/`, `client_delivery/`, `research_intelligence/`, and the RSI dept are structurally identical: load config → format prompt → `self.call_claude(...)` → `extract_json(resp.text)` → return `{confidence, result, tokens_in, cost_usd}`. This is the strongest single signal that the system was written to satisfy "26 agents" rather than to identify 26 distinct units of work.

**Fix:** Extract a `PromptAgent` base class that takes (system_prompt, max_tokens, tier, hitl_gate). Collapse the deterministic agents (sentinel, attribution, icp_scorer, gatekeeper, token_optimizer, observer) into utility functions invoked by their actual triggering agents.

### Goal 6 — Production Deployment Readiness: **FAIL**

`railway.toml` declares 3 services. Cold deploy will be fragile:

- **Startup validation is absent.** `core/settings.py:21-46` defines required keys (`anthropic_api_key`, `supabase_url`, `discord_bot_token`, etc.) as `str` with empty-string defaults. Nothing rejects empty strings at boot. First Claude call fails at runtime, not at deploy. `main.py:34` calls `require_langfuse()` first in lifespan — if that *does* enforce keys, all three services fail to boot without Langfuse. Either way, the failure mode is wrong: it should be deterministic and named.
- **Worker scheduling does not survive restarts.** `worker_main.py:38-48` is `while True: await observer.run(); await asyncio.sleep(3600)`. `worker_main.py:51-68` computes "next Monday 14:00 UTC" and sleeps. On Railway's `restartPolicyType: ON_FAILURE`, any crash loses the sleep timer. Two replicas would both fire.
- **Two services have no health check.** `railway.toml:23-29` (worker) and `:31-40` (bot) have no `healthcheckPath`. Railway can't auto-heal a hung worker. The hourly observer loop has no liveness signal.
- **No `.dockerignore`.** All three Dockerfiles use `COPY . /app`, dragging in `.git`, `__pycache__`, `dashboard/node_modules` (the dashboard is supposed to be out of scope but it's in the tree). Every change rebuilds from scratch.
- **HMAC verification is conditionally off.** `api/webhooks/router.py:164` skips signature checks if the bridge isn't configured. In production this is a security failure.

**Fix:** (1) Add a `validate_settings()` call as the first line of `main.py` lifespan and `worker_main.py:__main__`; fail-fast on empty required keys when `RUNTIME_ENV=prod`. (2) Add `/health` endpoints to worker and bot processes; wire them into `railway.toml`. (3) Add `.dockerignore` at repo root. (4) Replace `asyncio.sleep` schedulers with APScheduler + Postgres jobstore, OR delete `worker_main.py`'s scheduling entirely and rely on n8n (`rsi_weekly_trigger.json` already does this).

### Goal 7 — Speed as a First-Class Constraint: **FAIL**

The critical path `lead webhook → ICP scored → outreach drafted` is sequential and dominated by avoidable I/O.

Latency walk:

1. `POST /api/v1/webhooks/lead/intake` → `emit(LEAD_INTAKE)` → enters chain *in the HTTP request context*.
2. **ENRICH** (`departments/lead_gen/enrich.py:51-87`):
   - Line 57: `await _hunter_lookup(domain, api_key)` — 1 external HTTP, 10s timeout, **blocks**.
   - Line 64: `await self.call_claude(...)` — Claude SONNET, max_tokens=700.
   - These two awaits are **sequential**, not `asyncio.gather`.
3. **ICP_SCORER** (`departments/lead_gen/icp_scorer.py:96-127`): deterministic rules, <20 ms. (This is the one fast step.)
4. **OUTREACH** (`departments/lead_gen/outreach.py:40+`):
   - Line 45: config load via `load_resolved_config` — sync file I/O on every call.
   - Lines 48-50: read brand voice markdown — sync file I/O on every call (no cache).
   - Line 62: `await self.call_claude(...)` — Claude SONNET, max_tokens=600.

P50 estimate: **5–7 seconds** wall-clock, dominated by sequential Hunter + Claude in enrich and a second Claude call in outreach. Cold Hunter + cold Claude approaches 10 s, which is at or above many HTTP gateway timeouts.

Other speed hits found:
- `analyst.py:64` — Pinecone queried unconditionally on every run, even for trivially short subjects (~15 ms-2 s wasted).
- `librarian.py:93` — 12 RSS feeds gathered with 12-second per-feed timeout; no wall-clock cap on `gather` itself (worst case 144 s before the await returns).
- `attribution.py:26`, `competitor.py:54`, `analyst.py:42` — `httpx.AsyncClient` constructed inside each request (no pooled singleton); TCP and TLS handshake on every Supabase call.
- `outreach.py:48-50` — brand voice file re-read from disk on every outreach.

**Fix:** (1) `asyncio.gather(_hunter_lookup(...), _claude(...))` in enrich with `wait_for` cap of 4 s. (2) Cache loaded `ClientConfig` and brand voice in process memory with an mtime check. (3) Singleton `httpx.AsyncClient` at the module level for Supabase. (4) Move the whole `/webhooks/lead/intake` chain off the request thread — return 202, enqueue, complete via webhook callback to client systems.

## PART 2 — Code Slop Report

### Slop Pattern 1 — Placeholder Implementations

- **MEDIUM** `core/schemas/base.py:100-127` — `Trigger`, `FieldSpec`, `SloSpec` defined as full Pydantic models but **never read anywhere**. They round-trip from YAML to no-op. *Fix:* remove or wire them to a real consumer.
- **LOW** `core/runtime/langfuse_client.py:84` — `finally: pass` swallows flush errors. *Fix:* log at error.
- `NONE FOUND` for `raise NotImplementedError` / explicit `TODO` comments inside the 108 scope files. (Surprising, but the slop here takes other forms.)

### Slop Pattern 2 — Over-engineered Abstractions

- **MEDIUM** `core/agents/base_agent.py:172-175` — `_execute_with_parent` sets `self._lf_parent` on the instance for one downstream consumer (`self.call_claude`). A parameter would be clearer and would not leak state across concurrent invocations on the same agent instance. *Fix:* thread `langfuse_parent` through the call chain as an arg.
- **MEDIUM** `core/schemas/base.py` — `ImprovementProposal` dataclass holds metadata but no actionable patch payload; the actual content (`new_prompt`) is returned in a sibling dict. The dataclass is decorative. *Fix:* include `file_path` and `new_content` as required fields, or delete the wrapper.
- **LOW** `core/runtime/n8n_bridge.py` — bridge class wraps a single HTTP POST. A function would suffice. *Fix:* keep only if a real second implementation lands.
- **LOW** Per-agent `_default_spec()` methods scattered across all 26 agents duplicate the same dataclass instantiation. *Fix:* declare specs in YAML registry, hydrate centrally.

### Slop Pattern 3 — Duplicate Logic

- **CRITICAL** Identical `_execute()` shape across ≥11 files: `enrich.py:51`, `mapper.py:38`, `outreach.py:40`, `nurture.py:41`, `scribe.py:40`, `proposer.py:57`, `success_ops.py:40`, `librarian.py:79`, `competitor.py:87`, `analyst.py:60`, `strategist.py:40`. Each: load config → format prompt → `call_claude` → `extract_json` → return `{confidence, result, tokens_in, cost_usd}`. This is one agent rewritten 11 times. *Fix:* `PromptAgent` base class parameterized on `(system_prompt, max_tokens, tier, hitl_gate, output_schema)`.
- **HIGH** Supabase REST patterns repeated inline in `observer.py:21`, `token_optimizer.py:27`, `ux_reviewer.py:38`, `rag_auditor.py:71`, `attribution.py:21`, `competitor.py:54`, `analyst.py:42`. Five distinct callsites manually constructing the same `httpx.get(url, headers, params)` shape against PostgREST. *Fix:* extract a `core/adapters/supabase_repo.py` with typed methods.
- **MEDIUM** `_default_spec()` boilerplate duplicated across all 26 agents.

### Slop Pattern 4 — Fake Async

- **HIGH** `enrich.py:57` then `:64` — two awaits in sequence that are independent and should be `asyncio.gather`'d. Direct latency hit on the critical path.
- **MEDIUM** `librarian.py:93` — `asyncio.gather(*[_fetch_rss(u) for u in rss_urls])` has no outer `wait_for`; per-feed timeout multiplies if many feeds stall.
- **MEDIUM** `outreach.py:45-50` — sync file I/O (`load_resolved_config`, `brand_path.read_text`) called inside an async path on the critical lead-response chain.
- **MEDIUM** `worker_main.py:38-68` — `while True: await fn(); await asyncio.sleep(3600)` is not a scheduler. *Fix:* use APScheduler.
- `NONE FOUND` for `asyncio.run()` inside an async context.

### Slop Pattern 5 — Error Theater

- **HIGH** `core/runtime/event_bus.py:76-77` — `except Exception: pass` on event persistence with only a warning log; no retry. Telemetry can be lost without signal.
- **HIGH** `core/runtime/claude_router.py:155-156` — `gen.update()` failure (cost/latency metrics) silently dropped.
- **HIGH** `core/telemetry/langfuse_client.py:92-93` — `score()` errors silently swallowed.
- **HIGH** `enrich.py:40-41` — Hunter API failure returns `None` with no log; downstream confidence calculation actually *increases* confidence when Hunter returns no data (`enrich.py:76`), which is backwards.
- **MEDIUM** `competitor.py:77` — `log.warning(..., extra={competitor})` then proceeds as if snapshot saved successfully. Future diffs compare against a stale snapshot.
- **MEDIUM** `core/runtime/claude_router.py:121-130` — retry loop's `last_err` may be `None` after three attempts; final `raise last_err or RuntimeError(...)` can mask the true cause.

### Slop Pattern 6 — Confidence and HITL Theater

- **CRITICAL** `api/approvals/router.py:65-70` — the approval callback emits `CoreEventType.HITL_RESOLVED` and updates a Supabase row. The comment claims "LangGraph thread listens and continues from `interrupt()`", but **no LangGraph thread is registered as a listener, no checkpoint ID is stored in `ApprovalRequest`, no `PostgresSaver` is configured anywhere**. The originating workflow has no mechanism to wake up and read the decision. Approvals are recorded; nothing resumes. *Fix:* store thread_id + checkpoint_id with the approval, and on decide() call `graph.update_state(...)` + `graph.invoke(None, config={'configurable': {'thread_id': ...}})`. OR document that approvals are out-of-band and stop calling them HITL.
- **HIGH** Hardcoded confidence floors across the system: `observer.py:99` (0.95 always), `gatekeeper.py:68` (0.95 / 0.5 - 0.1·issues), `sentinel.py:71` (0.9 or 0.6), `rag_auditor.py:85` (default 0.7), `token_optimizer.py:98` (0.9 or 0.6), `icp_scorer.py:116` (`0.6 + 0.4·coverage`), `proposer.py`/`outreach.py` (Claude self-reports). The confidence engine consumes these without validating that they are even calibrated. *Fix:* either calibrate against outcome data (deal won/lost, PR merged/rejected) or stop calling them confidence and call them rule-flags.
- **HIGH** `proposer.py:47-55` — declares `HitlSpec(gates=[...])` in spec but `_execute()` completes synchronously and returns. No `interrupt()` call, no wait. The HITL gate is advisory metadata only.
- **MEDIUM** `core/runtime/confidence_engine.py:43-44` — per-agent threshold math (`min(self.low, max(0.0, spec.confidence_threshold - 0.25))`) can produce a `per_agent_low` below `per_agent_high` by a wide margin, putting scores into MEDIUM that should be HIGH. *Fix:* document or refactor.

### Slop Pattern 7 — Anthropic API Misuse

- **HIGH** `core/runtime/claude_router.py:64-73` — `_with_cache_control` applies `cache_control: ephemeral` only when the system prompt is a string; if a caller passes pre-blocked content, cache is silently dropped. Inconsistent cache hits across agents. *Fix:* unconditionally apply to text blocks.
- **MEDIUM** `core/runtime/claude_router.py:56-61` — `_model_for(tier)` falls through to SONNET on any unknown tier with no warning. *Fix:* dict lookup with explicit raise on missing key.
- **MEDIUM** No tier discipline: all "real" creative agents (PROMPT_OPTIMIZER, SEEKER, SYNTHESIS, OUTREACH, NURTURE) use SONNET regardless of task complexity. Cost-sensitive judgment tasks like UX_REVIEWER (`max_tokens=2000`) should be HAIKU first with SONNET fallback. *Fix:* tier-by-task table in `claude_router.py`.
- **LOW** `max_tokens` values appear chosen by feel: 2000 for UX review of 8 samples, 1500 for synthesis picking 3 items, 1200 for SEEKER finds. No estimation of upper bound. Not a bug, but no defense against runaway.
- `NONE FOUND` for leftover `user=` kwarg from Failure #2 (CLAUDE.md). That fix held.

### Slop Pattern 8 — Supabase & Pinecone Misuse

- **HIGH** `departments/research_intelligence/analyst.py:64` — Pinecone queried on every run regardless of whether RAG is needed for the subject. *Fix:* skip when `len(subject) < N` or when the agent's role flag says RAG-off.
- **MEDIUM** `attribution.py:21`, `observer.py:21`, etc. — REST queries with no `select=` parameter; PostgREST returns all columns. Wider tables = more bytes per query. *Fix:* explicit `select=col1,col2`.
- **MEDIUM** Multiple call sites instantiate `httpx.AsyncClient` inside the handler (no pooling) — `attribution.py:26`, `competitor.py:54`, `analyst.py:42`. *Fix:* module-level singleton with `AsyncClient(http2=True, limits=...)`.
- **MEDIUM** Hardcoded table paths sprinkled across agents (`/rest/v1/agent_runs`, `/rest/v1/events`, `/rest/v1/founder_review_queue`). A schema rename breaks N agents at once. *Fix:* repository module.
- **MEDIUM** `competitor.py:64-76` — always saves a new snapshot before comparing; if the save silently fails, the next run compares against the still-stale prior snapshot indefinitely.
- `NONE FOUND` for explicit `loop + upsert` pattern, but several agents call Supabase inside per-item iterations indirectly via shared helpers.

- **HIGH** `observer.py`, `token_optimizer.py`, `gatekeeper.py` make zero Claude calls. They are SQL/validation in agent costume.

- **HIGH** `api/webhooks/router.py:164` — HMAC verification is conditional on `bridge.configured`. Production deploy with empty `N8N_HMAC_SECRET` leaves callback endpoints open.

- **CRITICAL** `worker_main.py:51-68` and `n8n/recipes/rsi_weekly_trigger.json` both fire at Monday 14:00 UTC. No mutex, no env flag, no idempotency on the RSI loop. Approved PRs will be opened twice. *Fix:* `ENABLE_WORKER_SCHEDULER` env (default false). Single source of truth = n8n.

## PART 4 — What Should Have Been Built Differently

### Three biggest architectural mistakes

1. **HITL was specified before the resume mechanism existed.** Every agent that gates on confidence declares a `HitlSpec`, but no agent actually pauses. `api/approvals/router.py` writes a decision to a DB row and emits an event — there is no LangGraph checkpoint resume, no thread state, no polling loop in the workflow. Build the resume mechanism *first*, then layer HITL specs on top.
2. **RSI was modeled as 7 distinct agents instead of one pipeline.** Observer + token_optimizer are SQL rollups. UX_reviewer + RAG_auditor are LLM judges. Prompt_optimizer + synthesis + patcher form a real chain. Calling all seven "agents" muddied responsibilities and produced the schema mismatch between `prompt_optimizer` output and `patcher` input that silently drops most actions.
3. **Real-estate assumptions were baked into the loader, not the pack.** `config_loader.py:81-90` hardcodes the names of RE-specific sub-files. A clean factory would have the pack declare its own sub-files in a manifest and the loader treat them as opaque.

### Three places where complexity was added unnecessarily

1. **26 `_execute` methods** that are 90% the same. A `PromptAgent(system, max_tokens, tier, hitl_gate, output_schema)` would replace 11 of them outright.
2. **The Trigger/FieldSpec/SloSpec schema branches** in `core/schemas/base.py` (lines 100-127) round-trip from YAML and are read by nothing.
3. **Two RSI schedulers** — one in `worker_main.py`, one in `n8n/recipes/rsi_weekly_trigger.json`. Two sources of truth, no lock.

### Three things missing that will cause real problems within 30 days

1. **No startup validation of required env vars.** First real client deploy will silently launch with an empty `ANTHROPIC_API_KEY` or `SUPABASE_SERVICE_ROLE_KEY`; the first lead webhook will fail at runtime in front of the client.
2. **No webhook idempotency.** `tally_onboarding_intake.json` and `lead_sheet_sync.json` will retry on n8n's policy; the receiving endpoints do not deduplicate. A retried lead intake = a second Claude call + a second Supabase row.
3. **No GitHub post-merge hook → no config reload.** Even when RSI does open a PR and the user merges it, the running container won't pick it up until next deploy. The "self-improvement" never reaches production.

### Single most urgent fix before a real client

**Make HITL actually pause workflows, or remove the HITL spec from every agent.** Right now the system claims human approval gates that don't exist. A client will approve a proposal in Discord, the row will update in Supabase, and the downstream agent will already have continued. Either implement LangGraph thread/checkpoint resume on `/approvals/{id}/decide`, or be honest and label these "review queues" instead of "human-in-the-loop gates".
